## 4. Template Conversion — One-Time Step

The `TS_Template.docx` must be converted from `{{DoubleHandlebars}}` to docxtpl's Jinja2 syntax. Create `backend/scripts/convert_template.py`:

```python
INPUT  = Path("templates/TS_Template_original.docx")

def replace_in_paragraph(para):
    """Rebuild paragraph text with replacements applied, preserving run formatting."""
    full_text = "".join(run.text for run in para.runs)
    new_text = full_text
    for old, new in REPLACEMENTS.items():
        new_text = new_text.replace(old, new)
    if new_text != full_text and para.runs:
        para.runs[0].text = new_text
        for run in para.runs[1:]:
            run.text = ""

def process_document(doc):
    for para in doc.paragraphs:
        replace_in_paragraph(para)
    for table in doc.tables:
        for row in table.rows:
            for cell in row.cells:
                for para in cell.paragraphs:
                    replace_in_paragraph(para)

if __name__ == "__main__":
    doc = Document(str(INPUT))
    process_document(doc)
    doc.save(str(OUTPUT))
    print(f"Converted template saved to {OUTPUT}")
    print("IMPORTANT: Open TS_Template_jinja.docx and verify the features table")
    print("has {%tr for feature in features %} and {%tr endfor %} row-level tags.")
    print("See docxtpl docs: https://docxtpl.readthedocs.io/en/latest/#for-loop-in-a-table")
```

> **IMPORTANT — Features Table:** After running this script, open `TS_Template_jinja.docx` in Word and locate the Features section (3.1). The template has 6 feature rows in a table. docxtpl requires `{%tr for feature in features %}` at the START of the first feature row and `{%tr endfor %}` at the END of the last feature row — these go inside a table cell in the respective rows, at the XML level. Without this, the for-loop renders as literal text. See: https://docxtpl.readthedocs.io/en/latest/#for-loop-in-a-table
>
> **MVP fallback:** If the for-loop table fix is complex, use individual placeholders `feature_1_title` through `feature_6_title` (etc.) with a max of 6 features in the UI. Empty features render as blank strings. Document this as a known constraint.

## 7. API Endpoints — Complete Specification

### 7.1 Projects

| Method | Path | Description | Body | Response |
|--------|------|-------------|------|----------|
| `GET` | `/api/v1/projects` | List all projects | — | `[ProjectSummary]` |
| `POST` | `/api/v1/projects` | Create project | `ProjectCreate` | `ProjectDetail` |
| `GET` | `/api/v1/projects/{id}` | Get project + completion | — | `ProjectDetail` |
| `PATCH` | `/api/v1/projects/{id}` | Update global fields | `ProjectUpdate` | `ProjectDetail` |
| `DELETE` | `/api/v1/projects/{id}` | Delete project & sections | — | `204` |

**`ProjectCreate` / `ProjectUpdate` fields:**
```
solution_name, solution_full_name, solution_abbreviation,
client_name, client_location, client_abbreviation,
ref_number, doc_date, doc_version
```

**`ProjectDetail` response** includes all project fields plus:
```json
{
  "completion_summary": { "total": 27, "completed": 14, "percentage": 51 },
  "section_completion": {
    "cover": true, "revision_history": false, "features": true, ...
  }
}
```

### 7.2 Section Data

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/projects/{id}/sections` | All section data |
| `GET` | `/api/v1/projects/{id}/sections/{section_key}` | Single section |
| `PUT` | `/api/v1/projects/{id}/sections/{section_key}` | Upsert section content |

**`backend/app/generation/docx_generator.py`:**
```python
from docxtpl import DocxTemplate
from pathlib import Path
from app.generation.context_builder import build_context


def generate_document(project, all_sections: dict, template_path: str, upload_dir: str) -> str:
    """Render template, save to disk, return file path."""
    template = DocxTemplate(template_path)
    context  = build_context(project, all_sections, template, upload_dir)
    template.render(context)

    output_dir = Path(upload_dir) / "versions" / project.id
    output_dir.mkdir(parents=True, exist_ok=True)

    safe_client   = project.client_name.replace(" ", "_").replace("/", "-")[:30]
    safe_solution = project.solution_name.replace(" ", "_")[:20]
    filename      = f"TS_{safe_client}_{safe_solution}_v{project.doc_version or 0}.docx"
    output_path   = output_dir / filename

    template.save(str(output_path))
    return str(output_path), filename
```

**`frontend/src/hooks/useAutoSave.ts`:**
```typescript
import { useCallback, useRef, useState } from 'react'
import { updateSection } from '../api/sections'

type SaveStatus = 'idle' | 'saving' | 'saved' | 'error'

export function useAutoSave(projectId: string, sectionKey: string, delay = 800) {
  const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null)
  const [status, setStatus] = useState<SaveStatus>('idle')

  const save = useCallback((content: object) => {
    if (timerRef.current) clearTimeout(timerRef.current)
    setStatus('saving')
    timerRef.current = setTimeout(async () => {
      try {
        await updateSection(projectId, sectionKey, content)
        setStatus('saved')
        setTimeout(() => setStatus('idle'), 2000)
      } catch {
        setStatus('error')
      }
    }, delay)
  }, [projectId, sectionKey, delay])

  return { save, status }
}
```

## 19. Critical Implementation Notes

These are non-negotiable points that Kiro must implement exactly as described.

**1. SolutionName global auto-fill is non-negotiable.**
Every section component must read `solutionName` from `useProjectStore`. Anywhere in the UI where the template shows `{{SolutionName}}` — section titles, table row labels, preview text, locked section bodies — render the live value from the store. Do not use a static string. When the user changes the solution name on the Cover section, it propagates instantly to all open section previews without a page refresh. The store value is set when the project loads and when the Cover section saves.

**2. Generate button triggers a file download — not navigation.**
The `POST /api/v1/projects/{id}/generate` endpoint returns a `FileResponse`. The frontend must handle this as a binary blob download:
```typescript
const response = await client.post(`/api/v1/projects/${id}/generate`, {}, { responseType: 'blob' })
const url = window.URL.createObjectURL(new Blob([response.data]))
const link = document.createElement('a')
link.href = url
link.setAttribute('download', `TS_${projectId}.docx`)
document.body.appendChild(link)
link.click()
link.remove()
```

**3. Auto-save must not block UI.**
The `useAutoSave` hook fires a debounced save 800ms after the last keystroke. The "Saved ✓" indicator appears 2 seconds then fades. This runs entirely in the background — no loading spinners, no disabled inputs, no blocking.

**4. Sections with only locked content still need a `SectionData` row.**
When the user first opens a locked section (Binding Conditions, Cybersecurity, Disclaimer), create a `SectionData` row with `content: {}` to mark it as "visited." This enables the completion calculation to see them as complete without any user action.

**5. Rich text stored as HTML, rendered as plain text in docxtpl.**
Tiptap produces HTML like `<p>Some text</p><ul><li>Item</li></ul>`. When building the docxtpl context, strip HTML tags to plain text using Python's `html.parser`. For the MVP, formatting inside rich text sections (bold, lists) will not appear in the generated Word doc — only the plain text content. This is an acceptable MVP constraint; document it clearly.

**6. The `generate` endpoint must handle missing images gracefully.**
If architecture diagram or either Gantt chart image has not been uploaded, the context builder places a plain-text placeholder string `"[Diagram — To Be Inserted]"` at that position in the context. docxtpl will render this as plain text in the Word doc. The generation must NOT fail because an image is missing — it should succeed and produce a complete document with placeholder text.

**7. Version numbering.**
Each call to `POST /generate` increments the version number. Query `MAX(version_number)` from `document_versions` where `project_id = ?`. If no versions exist, start at 1. The filename includes the version: `TS_ClientName_SolutionName_v1.docx`.

**8. No authentication, no protected routes, no user tables.**
Remove all auth completely. The app opens directly at `/` (project list). There are no login pages, JWT tokens, HTTP auth headers, or user foreign keys on any table. The `projects` and `section_data` tables have no `created_by` or `user_id` columns.

**9. Validate before generating — return clear errors.**
Before generating, check all required sections (§13 table). If any are incomplete, return HTTP 422:
```json
{
  "detail": "Cannot generate: required sections incomplete",
  "missing_sections": ["process_flow", "fat_condition", "supervisors"]
}
```
The frontend shows these as a list of clickable section names: "Please complete these sections first: Process Flow, FAT Condition, Supervisors." Each is a link that navigates directly to that section in the sidebar.

**10. `{%tr for %}` loop for features — do not skip this.**
docxtpl's table row loop syntax requires `{%tr for feature in features %}` inside the first repeating row's first cell, and `{%tr endfor %}` inside the last repeating row's last cell — at the XML level in the `.docx` template. Without this, the for-loop tag renders as literal text in every feature row. After running `convert_template.py`, open `TS_Template_jinja.docx` and manually add these XML tags. If blocked, use the fallback of 6 fixed feature variables (`feature_1_title` through `feature_6_*`) and document it.
