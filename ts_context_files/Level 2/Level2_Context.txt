# LEVEL 2 DOMAIN KNOWLEDGE

## What Level 2 PMS Is
A production management system (PMS) that sits between Level-1 plant-floor control (PLC/HMI) and Level-3 enterprise systems (ERP/MES/business systems). It orchestrates manufacturing execution for one or more production lines: receives production-unit specifications, converts them into machine setpoints, tracks the unit of production through the line, captures actual results, and exchanges this data with L1 and L3. NOT a SCADA or visualization-only tool, NOT the L1 control system itself, and NOT a data-acquisition/historian middleware platform — that is a distinct product category that may sit alongside or beneath an L2 PMS but serves bulk tag collection and redistribution rather than production-order orchestration.

## Core Capabilities (present in every L2 PMS deployment)
- Primary Data Input (PDI): receives production-unit specification data from L3 (online) or manual HMI entry before the unit reaches the line; validates against upper/lower limits and data congruency; alarms the operator on bad data; supports sequence changes and manual override when no L3 system exists
- Setup/schedule function: converts PDI into machine reference setpoints using existing recipe-based look-up tables/algorithms (buyer-provided); outputs values such as speed, tension, reduction, force, and power references for L1
- Adaptive/learning function (where the process supports it): compares calculated setpoints against actual process results, derives compensation coefficients, and updates the look-up-table-driven schedule over time; scope (which parameters are covered) is project-dependent and may be limited to a single parameter unless commercially extended
- Production Data Output (PDO) / delivery file control: captures final production-unit results at the unload/completion event and sends them to L3
- Equipment/tooling management (e.g., roll, tool, die data): registers and tracks current vs. next equipment in use, cumulative usage, and event-driven transitions coordinated with L1 (insertion/change events) and L3/asset systems
- Downtime management: captures and classifies line stoppage events
- Reporting function: production and performance reports from collected process data
- Data link with L1 PLC (read/write setpoints and status) and data link with L3 computer (PDI in, PDO out)

## Optional/Value-Added Capabilities (project-dependent)
- Mobile/HHT application for executives/supervisors to view live and historical process data with appropriate access control
- Data visualization and analytics dashboards (customized per buyer requirement during engineering)
- SMS alert system for significant process/line events
- Email alert and reporting system delivering production reports/logs at a predefined frequency
- Remote post-commissioning support (duration project-dependent)
- On-site customer training (duration and trainee count project-dependent)

## Business Drivers
Customers implement an L2 PMS to: move from manual/paper or siloed production tracking to fact-based decision making, give line managers and operations supervisors live and historical visibility into process performance, fine-tune process setpoints beyond static recipe tables, establish traceability for each production unit, and lay the groundwork for further digitalization (centralized dashboards, AI/analytics, scheduling tools). An L2 PMS never guarantees productivity, quality, or yield outcomes.

## Plant Environment
Discrete or semi-continuous manufacturing with one or more production lines/units, each with its own existing L1 system (PLC/HMI; make and model are project-dependent — do not assume). An existing L3/ERP system may or may not be present; if absent, the L2 PMS provides manual PDI entry and production-sequence entry as a fallback. OT network is separated from IT; firewalls are buyer scope.

## Architecture Options
Three-tier topology: L1 (PLC/HMI per production line) → L2 (PMS application server(s), client workstations) → L3 (ERP/business systems), connected via managed switch and buyer-controlled firewall.
High-availability option: online/primary server plus backup/standby server with a heartbeat connection; client workstations connect to whichever server is active. Use only when project metadata confirms HA is in scope.
Remote support access is via a buyer-provisioned internet line/VPN to a seller remote-support location; never assume default connectivity exists.

## Protocols
L1↔L2: project-dependent ([PROTOCOL] — examples observed: vendor-proprietary PLC link, TCP/IP). Confirmed and finalized during detailed engineering, not at TS stage.
L2↔L3: project-dependent (examples observed: DB-LINK, TCP/IP). If an L3 system is absent, this interface does not apply.
Do not assume OPC UA, Modbus, or any specific protocol unless project metadata confirms it — L2 PMS protocol needs differ from data-acquisition-middleware protocol needs (which favor OPC UA/Modbus/MQTT).

## Data Acquisition
The L2 PMS receives PDI from L3 (or manual entry) and process data from L1 in real time during production. It is not a bulk multi-protocol tag-polling platform; tag/point counts when stated ([NUM_TAGS]) describe data passed through an adjoining data-acquisition layer, not the L2 PMS application itself — do not conflate the two.

## Data Storage
On-premises database (existing plant database may be reused, or a new instance provisioned — [DB_PLATFORM] from metadata, do not assume a specific RDBMS). Retention of production-unit history is configurable and project-dependent ([RETENTION_PERIOD] — observed examples only, not a default).

## Integration Patterns
1. L3 → L2 (PDI inbound): buyer provides the production-order/specification data feed; seller configures the L2-side ingestion.
2. L2 → L1 (setpoints/setup outbound): seller configures the L2-side; buyer's existing L1 system must be able to accept the interface.
3. L1 → L2 (process data/results inbound): buyer ensures L1 exposes required signals; seller configures the L2-side acquisition.
4. L2 → L3 (PDO outbound): seller configures the L2-side; buyer's L3 system must be able to receive the interface.
5. L2 ↔ adjoining data-acquisition/historian layer (if present): seller may reuse a separate concentrator/gateway product for bulk L1 tag collection feeding L2 dashboards; this is architecturally distinct from PDI/PDO production-order exchange.
All interface parameters (addressing, schema, protocol specifics) are finalized during detailed engineering — not at TS stage.

## Technology Stack Patterns (observed, not defaults)
Software: web-based frontend framework, Python/Django or equivalent backend framework, containerized deployment, existing or new RDBMS, Windows Server OS, enterprise antivirus. All versions: "current at detailed engineering start."
Hardware: enterprise tower/rack server (sizing determined by number of lines, tag volume, and retention — never state specific RAM/storage values), FHD client console monitors, client desktop PCs, managed/unmanaged network switches, media converters for fiber-to-Ethernet where applicable, GSM modem for SMS alerting (if in scope).
Buyer provides: server room, power, network cabling and accessories beyond core scope, firewall, SIM for the GSM modem, access to existing recipe/look-up tables.

## Cybersecurity
Buyer owns: all cybersecurity, information security, hardware/network/data security, patch management post-delivery, firewall configuration, VPN/remote-access link management.
Seller does: delivers the application; recommends prompt patching and use of supported software versions.
Key TS statements: seller not liable for security incidents, data breaches, or unauthorized access; NDA required before remote support sessions begin.
Do not generate novel legal security clauses — use approved TS boilerplate.

## Implementation Phases
Formal order acceptance (M0) → System design (M1: design document approval) → Hardware engineering and software design (parallel) → Software development — programming and testing (M2) → Shop test/FAT at seller facility (M3) → Software shipping → Site commissioning: deployment, communication testing with L1 and L3, reporting/alert configuration, backup, go-live, overall test, monitoring, commissioning report (M4/M5: work completion).
Critical path: design document approval and timely delivery of buyer-side interface data (recipe tables, L1/L3 schema) are the most common sources of delay — both buyer-gated.
On-site man-days: project-dependent, agreed commercially; travel days excluded.

## Testing
FAT/shop test (seller facility): validates configured PDI/PDO flows, setup/adaptive logic (if in scope), and reporting against simulated or representative data; sign-off gates shipment.
Commissioning/SAT (site): live interface test with L1, live interface test with L3 (if present), reporting and email/SMS alert verification, backup verification, go-live, monitoring period, commissioning report.
Acceptance artifacts: FAT/shop test result, commissioning report, work completion certificate, functional specification, hardware specification, software specification, system interface document.

## Risks
1. Buyer's recipe/look-up table data, procedures, or algorithms not provided or incomplete — impairs setup/adaptive accuracy; owner: Buyer
2. L3 PDI feed delayed, malformed, or absent — forces fallback to manual entry, delaying integration testing; owner: Buyer
3. L1/L3 interface protocol and parameters not finalized in time for detailed design; owner: Shared
4. Existing database/L3 schema undocumented or not provided; owner: Buyer
5. Adaptive/learning function scope misunderstood as covering all process parameters when commercially limited to one; owner: Shared
6. Network/VPN/internet connectivity not ready at commissioning; owner: Buyer
7. Shutdown window insufficient for cutover, communication testing, and go-live activities; owner: Shared
8. Buyer's third-party OEM/equipment engineers unavailable when needed for engineering or commissioning; owner: Buyer
9. Scope creep from undocumented or additional production lines beyond those named in the TS; owner: Shared
10. Buyer does not execute post-handover network/cybersecurity configuration responsibilities; owner: Buyer

## Standard Assumptions
- Buyer provides accurate, complete plant technical data (line specifications, products, existing automation) ahead of design
- Buyer's existing L1 system can expose the required interface; if not, upgrade is buyer scope
- Buyer provides existing recipe/look-up tables, mathematical models, and algorithms needed for setup and adaptive functions
- L1↔L2 and L2↔L3 interface protocol and parameters are finalized during detailed engineering, not at TS stage
- Buyer network provides adequate bandwidth and routing for L1/L3 interfaces and remote support
- Named production lines/units constitute complete scope; additional lines are a change order
- If an existing database is to be reused, it is available and accessible for L2 integration

## Standard Exclusions
- Interfaces with systems not explicitly named in the TS
- OS patches, antivirus updates, and other software maintenance after delivery
- Source code of the core L2 platform
- Modifications to existing L1, legacy L2, MES, or L3/ERP systems
- Erection activities and civil work
- Mechanical equipment supply or modification
- Troubleshooting of mechanical, operational, or process issues unrelated to the L2 application
- Data-acquisition/historian middleware software and hardware, where positioned as a separate product
- Firewalls and other networking infrastructure
- Performance guarantees regarding productivity, quality, yield, or process capability

## Diagram Guidance (semantic only — no mxGraph XML)
Architecture layers (top to bottom): L3 Layer (ERP/business systems, firewall — buyer scope) → L2 Layer (online server + optional backup server with heartbeat, client workstations) → L1 Layer (PLC/HMI per production line) → Field.
Connection labels: L1↔L2 = [PROTOCOL]; L2↔L3 = DB-LINK/TCP-IP (if L3 present); online↔backup server = heartbeat; remote support link = Internet/VPN.
Scope boundaries: dashed = seller scope, alternate dashed = buyer scope. Legend: connection type key, scope boundary key, reference-only caveat.
Data flow: L3 → PDI → L2 setup/adaptive function → setpoints → L1 → process results → L2 → PDO → L3. Annotate arrows with direction and data type (PDI/PDO/setpoints/results).
Note: where a separate data-acquisition/historian layer feeds dashboards, show it as an adjoining, not nested, block — it is architecturally distinct from the PDI/PDO exchange.

## Gantt Guidance (all values indicative only)
Phases: Order Acceptance (M0) → System Design (M1: design approval) → Hardware Engineering / Software Design (parallel) → Software Development — Programming and Testing (M2) → Shop Test/FAT (M3) → Software Shipping → Commissioning (M4) → Work Completion (M5).
Scale examples (observed, not commitments): single-line, no L3 integration tends to run shorter; multi-line with L3 integration and HA tends to run longer. Do not promote any specific week/month count as standard.
Draw.io format: week-based JSON tasks with fields: task name, phase, start_week (integer, week 1 = PO acceptance), duration_weeks (integer), milestone (bool), dependencies (array of predecessor indices). Backend converts to mxGraph XML.
Shutdown Gantt: applicable when commissioning cutover requires a production shutdown window. Structure as Before-MSD (preparation, deployment) / Main Shutdown (communication tests with L1 and L3, reporting/alert setup, backup) / After-MSD (go-live, overall test, monitoring, commissioning report), with day offsets relative to shutdown start, not calendar dates.

Required: Platform name(s) and type (from metadata); demonstration period; hosting arrangement (typically on the L2 server infrastructure).
Infer: Complimentary, time-limited, hosted on existing L2 server infrastructure, does not constitute a deployment commitment, IP remains with seller, buyer must purchase via separate commercial contract to continue use, seller may withdraw if buyer does not proceed.
Never invent: Platform capabilities not listed in scope; deployment timelines; commercial terms.
Structure: One short subsection per PoC platform. State what the platform demonstrates, where it runs, the demonstration duration, and the continuation/withdrawal terms. Explicit scope-limiting language required.

## tech_stack

Required: All components from project metadata or standard L2 PMS BOM/stack. Backup-server line item if HA is in scope.
Infer: Standard software and hardware patterns (see Technology Stack Patterns in domain knowledge above).

## binding_conditions
Role: Contractual framework limiting seller scope and establishing seller site-exit rights.
Required: Nothing project-specific — standard content only.
Infer: Scope limited to TS-defined content; services outside = buyer responsibility; no performance guarantees; conditions for site exit (idling, buyer non-compliance, force majeure, unmet prerequisites, non-conducive site circumstances); no liability for equipment damage unless stated; no responsibility for system modification unless stated; contract effective date = PO acceptance date.
Never invent: Novel legal clauses; penalty or indemnity terms.
Structure: Bulleted list. Use repository-approved binding conditions boilerplate — do not generate novel legal language.
