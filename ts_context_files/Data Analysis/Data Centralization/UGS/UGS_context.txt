# UGS DOMAIN KNOWLEDGE

## What UGS Is
Industrial data integration middleware: collects process data from multiple heterogeneous plant automation systems, normalizes it, stores it centrally, and redistributes it to higher-level consumers. Sits at the OT/IT boundary. NOT a SCADA, visualization tool, or process controller.

## Core Capabilities (all UGS deployments)
- Centralized data acquisition from multiple production units via configurable polling
- Multi-protocol: OPC UA (primary when confirmed), Modbus, OPC DA, TCP/IP, UDP, MQTT, DBlink/SQL
- Browser-based web configuration UI — no custom programming for standard setup
- On-premises SQL database with configurable FIFO retention and automated housekeeping
- OPC UA server role exposing acquired data to downstream consumers
- Role-based user accounts, full user action audit trail
- Email alerting/reporting (value-added, project-dependent)
- Remote support post-commissioning (value-added, duration project-dependent)
- On-site training at commissioning (duration and count project-dependent)

## Optional: HX Edge Controller
Installed at each production unit. Provides local buffering, preprocessing, protocol bridging. Prevents data loss during network disruptions. Use only when direct central polling is insufficient.

## Business Drivers
Customers implement UGS to: eliminate siloed data per unit, replace point-to-point integrations, enable plant-wide KPI visibility, create digitalization foundation (MES, analytics, AI/ML, cloud), reduce manual reporting, apply consistent data governance.
UGS never guarantees production performance, product quality, or yield.

## Plant Environment
Continuous-process manufacturing, multiple production units with existing PLCs, HMIs, and data acquisition servers (make and presence are project-dependent — do not assume). OT network (L1 LAN, L2 LAN) separated from IT. OT firewalls always buyer scope.
UGS integration is strictly read-only. It cannot configure existing L1/L2 systems — buyer or buyer's OEM must do that.

## Architecture Options
Option A (Direct): UGS server polls each unit's data acquisition server centrally. No edge hardware. Use when network reliability is high.
Option B (HX per unit): HX edge controller at each unit → central UGS server. Use when network reliability cannot be guaranteed.
Layers (both): Field → L1 (PLCs, PDA servers, L1 LAN) → L2 (supervisory, L2 LAN) → OT Boundary (firewalls, buyer scope) → IT (UGS server, managed switch, downstream consumers) → Remote Access (buyer-controlled VPN).

## Protocols
Primary: OPC UA. Also: OPC DA, Modbus RTU/TCP, TCP/IP raw, UDP, MQTT, DBlink/SQL.
Protocol selection is project-dependent. Outbound: UGS OPC UA server for live subscriptions; SQL for batch (heavy queries degrade acquisition performance — discourage).

## Data Acquisition
Configurable polling tiers (all values project-dependent — observed examples only, not defaults):
- Real-time: frequent polling of critical tags
- Supervisory: less frequent, L2 databases or aggregation servers
- High-frequency (optional): sub-second for AI/ML or diagnostics
Tag capacity architecturally scalable. [NUM_TAGS] from project metadata. Do not assume a number.

## Data Storage
Centralized on-premises SQL database. Retention: configurable FIFO ([RETENTION_PERIOD] from metadata — do not assume 1 year). Automated housekeeping prevents manual DBA effort. No write-back to source systems.

## Integration Patterns
1. OPC UA Client → data acquisition server (IBA PDA or equivalent if confirmed): buyer configures OPC UA server side; seller configures UGS client side.
2. OPC UA or Modbus → PLC direct: buyer provides PLC details and addressing.
3. DBlink/SQL → L2 database: buyer provides schema, read-only credentials, network route.
4. UGS OPC UA server → downstream consumers (analytics, FTDM, MES): buyer configures downstream client.
All interface parameters finalized during detailed engineering — not at TS stage.

## Technology Stack Patterns (observed, not defaults)
Software: browser-based frontend, C#/.NET Core backend, SQL Server, Windows Server, enterprise antivirus. All versions: "current at detailed engineering start."
Hardware: enterprise tower/rack server (sizing determined by tag count, polling frequency, retention period — never state specific RAM/storage values), FHD monitor, HIL-approved managed switch, network cabling (length per site), HX controller + junction box (if Option B).
Buyer provides: server room, power, firewall, SMS gateway (if needed), erection materials.

## Cybersecurity
Buyer owns: all cybersecurity, information security, hardware/network/data security, patch management post-delivery, VPN/remote access link management.
Seller does: delivers secure-by-default configuration, antivirus at delivery, recommends prompt patching.
Key TS statements: seller not liable for security incidents; NDA required before remote sessions; buyer controls VPN.
Secure defaults: RBAC active at deployment, default credentials changed at commissioning, all actions audit-logged, SQL access read-only for external consumers.
Do not generate novel legal security clauses — use approved TS boilerplate.

## Implementation Phases
Initiation → Pre-Engineering (buyer provides tag lists/docs, M1 gate) → Basic Engineering (system design + buyer approval, M2 gate) → Detailed Design (config, tag mapping, HX if Option B) → FAT (seller facility, M3 gate) → Shipping → Site Preparation/buyer parallel (M4 gate) → Commissioning (install, integrate, test, train, snag, handover) → Handover (docs, work completion cert, remote support starts).
Critical path: M1 (tag list delivery) and M2 (design approval) are the most common delays. Both are buyer-gated.
On-site man-days: project-dependent, agreed commercially. Travel excluded.

## Testing
FAT (seller): all configured flows against simulated sources, security config review, FAT report → gates shipment.
SAT/Commissioning (site): live acquisition per unit, OPC UA server test with real consumer, email via buyer SMTP, user accounts, documentation check.
UGS is read-only — removing it has no impact on plant L1/L2 operations.
Acceptance artifacts: FAT report, commissioning checklist, work completion cert, system operation doc, hardware spec, interface doc.

## Risks
1. Buyer tag list delay — most critical; define as formal prerequisite with deadline; owner: Buyer
2. Design approval delay — define turnaround in TS; owner: Buyer
3. Incompatible/undocumented source system — mitigate via pre-engineering survey; owner: Shared
4. Network not ready at commissioning — pre-visit checklist; owner: Buyer
5. OT firewall blocking traffic — document required ports in interface doc; owner: Buyer
6. Third-party OEM unavailable — state as buyer obligation; owner: Buyer
7. Scope creep from undiscovered units — define by named units; extras = change order; owner: Shared
8. Remote access link not established — buyer obligation, NDA required; owner: Buyer
9. Heavy SQL queries degrading concentrator — recommend OPC UA for real-time consumers; owner: Buyer
10. Security incident — covered by disclaimer; buyer owns network security; owner: Buyer

## Standard Assumptions
- Source systems can expose required interface; upgrades for that are buyer scope
- Tag list, data format, communication parameters finalized in detailed engineering
- Buyer network provides adequate bandwidth and routing
- Named units = complete scope; additional units = change order
- IP addresses and domain details provided by buyer during engineering

## Standard Exclusions
- Interfaces with systems not named in TS
- OS patches, antivirus updates, software maintenance after delivery
- Source code of core UGS platform
- Modifications to existing L1, L2, MES, ERP, automation systems
- Erection and civil works
- Firewalls and networking infrastructure
- PDA software and PDA hardware
- Mechanical equipment
- Performance guarantees on productivity, quality, yield, or process capability
- Supervision of buyer personnel, contractors, agents

## Diagram Guidance (semantic only — no mxGraph XML)
Architecture layers (top to bottom): Remote Support (buyer-controlled VPN) → IT Layer (UGS server, managed switch, FTDM/analytics clients) → OT/IT Boundary (OT firewalls, buyer scope) → L2 Layer (L2 servers per unit) → L1 Layer (PDA servers, PLCs per unit) → [HX Layer, Option B only] → Field.
Connection labels: source→HX or source→switch=[PROTOCOL]; HX→switch=Ethernet; UGS→FTDM=OPC UA; UGS→analytics=OPC UA/SQL; remote link=VPN/Broadband.
Scope boundaries: dashed = seller scope, alternate dashed = buyer scope. Legend: connection type key, scope boundary key, reference-only caveat.
Data flow: source systems → [HX] → switch → UGS concentrator → OPC UA server → downstream. Annotate arrows with protocol and frequency tier.
Network topology: two zones (OT: L1+L2 LAN; IT: L3 + UGS server). OT firewalls at boundary. No IP addresses, port numbers, or VLANs — finalized in engineering.

## Gantt Guidance (all values indicative only)
Phases: Initiation → Pre-Engineering (M1: buyer tag list) → Basic Engineering (M2: buyer design approval) → Detailed Design → FAT (M3: FAT sign-off) → Shipping → Site Preparation (M4: site ready) → Commissioning → Handover (M5: work completion cert).
Scale examples (observed, not commitments): small project (few units, no HX) ≈10–14 weeks; medium (several units, HX) ≈14–20 weeks; large (many units, complex) ≈18–26 weeks.
Draw.io format: week-based JSON tasks with fields: task name, phase, start_week (integer, week 1 = PO acceptance), duration_weeks (integer), milestone (bool), dependencies (array of predecessor indices). Backend converts to mxGraph XML.
Shutdown Gantt: UGS commissioning is typically non-intrusive — no full shutdown required. Brief panel/cable access managed as permits-to-work. If metadata confirms a shutdown, generate week-based tasks for shutdown-window activities only.
